## 문제 정의

- 게임 개발에서 `분석`과 `적용` 사이의 시간이 너무 길다.
- 개발자와 기획자가 테스트를 반복할수록 실제 유저와 다른 감각을 가지게 된다.
- 오픈 이후 난이도는 개발자 감각이 아니라 유저 행동 기준으로 다시 맞춰야 한다.

## 시스템 구조

- Base System:
  전체 난이도와 스테이지 난이도를 상수와 레벨 기반으로 분리
- Runtime Adjustment:
  Google Analytics와 스프레드시트로 클리어율, 진입 레벨, 기획 가중치를 관찰
- 운영 방식:
  스테이지 시작 시 서버에서 난이도 관련 값만 읽고 조정 가능

## 설계 포인트

- 초기 프로토타입 단계에서도 난이도 조정이 빠르도록 구조를 먼저 만든다.
- 레벨과 가중치만으로 스테이지 난이도를 바꿀 수 있게 단순화한다.
- 65~70% 정도의 클리어율 목표를 두고 실제 유저 로그로 계속 보정한다.
- `개발자 판단 -> 유저 행동 기반 보정`으로 난이도 조정 권한의 중심을 옮긴다.

## 이 문서가 보여주는 강점

- 로그 설계와 난이도 시스템 설계를 함께 보는 시야
- 기획 의도와 실제 플레이 데이터를 연결하는 방식
- 구현 복잡도를 낮추면서 운영 조정 속도를 높이려는 성향
- Google Analytics와 스프레드시트를 이용해 분석 접근 비용을 낮추는 실용주의
