# Inbox To Wiki Compiler

이 문서는 `Inbox`에서 `Wiki`와 `site`로 지식이 이동하는 전체 컴파일 규칙을 설명한다.

## 목적

- 원본 기록을 잃지 않으면서 자기지식을 축적한다.
- 매일의 메모를 장기적인 자기지식과 공개 가능한 프로필로 분리해 컴파일한다.
- 시간이 갈수록 `나에 대한 더 정확한 맵`이 만들어지게 한다.
- 같은 생각이 여러 번 나타날 때 하나의 노드로 합친다.

## 컴파일 모델

`Inbox`는 소스 코드다.  
`Daily`는 빌드 중간 산출물이다.  
`Wiki`는 누적되는 지식 베이스다.

그리고 `site`는 공개 가능한 항목만 다시 뽑아 만든 배포 산출물이다.

즉, 이 저장소는 단순 보관함이 아니라 `원본 -> 자기지식 -> 공개 프로필`로 이어지는 컴파일 시스템으로 본다.

즉, 모든 기록을 그대로 위키에 옮기는 게 아니라 아래를 골라낸다.

- 반복되는 자기서술
- 중요한 욕구와 가치
- 행동 패턴과 사고 패턴
- 반복되는 충돌과 한계
- 계속 남는 질문
- 사람, 사건, 프로젝트로 이어지는 단서

공개 페이지는 다시 한 번 더 거른다. `Wiki` 전체를 공개하지 않고, 경력,
강연, 선택된 프로젝트, 작업 방식처럼 외부에 보여도 되는 항목만 HTML로
정리한다.

## 비목표

- 모든 기록을 빠짐없이 `Wiki`에 옮기는 것
- 해석 문장을 사실 문장처럼 굳히는 것
- 내부 지식 베이스를 공개 페이지로 그대로 노출하는 것

## 기본 출력 경로

- 자기서술:
  `Wiki/Self/Profile.md`
- 최근 상태 진단:
  `Wiki/Self/Current Diagnosis.md`
- 다음 행동과 후속 TODO:
  `Wiki/Self/TODO.md`
- 자기 안의 충돌:
  `Wiki/Self/Tensions.md`
- 아직 답을 못 낸 질문:
  `Wiki/Self/Open Questions.md`
- 주제별 반복 패턴:
  `Wiki/Themes/*.md`
- 공부한 개념 정리와 확장 질문:
  `Wiki/Questions/*.md`
- 공개용 프로필:
  `site/index.html`

## Public Profile Compile

공개 프로필은 `Wiki -> site`로 한 번 더 컴파일하는 단계다.

### 공개 프로필 소스

- `Wiki/Self/Profile.md`
- `Wiki/Self/Career.md`
- `Wiki/Self/Speaking.md`
- `Wiki/Timeline/커리어.md`
- 공개 가능한 `Wiki/Projects/*.md`

### 공개 프로필에 넣는 것

- 비교적 안정적인 직업 정체성
- 공개 이력서에 이미 들어간 대표 성과
- 외부 발표와 강연 이력
- 대외적으로 보여도 되는 프로젝트
- 작업 방식과 반복되는 강점

### 공개 프로필에서 빼는 것

- 원본 `Inbox` 파일
- `Daily` 전체
- 가족과 아이 관련 세부 내용
- 건강 기록과 불안 기록
- `Tensions`, `Open Questions`처럼 사적인 내적 충돌
- 버전 차이가 정리되지 않은 민감한 사실
- 전화번호 같은 직접 공개 기본값이 아닌 연락처

### 공개 프로필 문장 원칙

- 이력서처럼 짧고 명확하게 쓴다.
- 확정이 어려운 정보는 단정하지 않고 완화해서 쓴다.
- 내부 위키의 해석 문장을 그대로 복붙하지 않는다.
- 공개용 요약이라는 사실을 페이지 어딘가에 남긴다.

## Wording Conflict Handling

같은 의미를 여러 워딩으로 쓰는 문제는 `컴파일 충돌`로 본다.

### 기준 문서

- 사용자가 직접 수정하는 승인 파일:
  `Meta/wording-conflicts.md`
- 정규 표현 사전:
  `Meta/wording-map.yaml`
- 정규 표현 사전은 `wording-conflicts.md`를 읽고 갱신하는 결과물로 본다.

### 컴파일 단계 안에서의 위치

1. 원본에서 사실, 자기서술, 링크, 질문을 추출한다.
2. 대상 문서를 갱신할 후보 문장을 만든다.
3. 같은 의미의 다른 워딩이 있는지 확인한다.
4. 있으면 `wording-conflicts.md`의 해당 항목을 본다.
5. `approved`면 `selected`를 사용한다.
6. `keep_distinct`면 합치지 않는다.
7. `open`이면 경고 상태로 컴파일을 계속한다.

### 처리 규칙

1. 새 문장을 쓰기 전에 `wording-conflicts.md`의 기존 항목이 있는지 확인한다.
2. `status: approved`면 `selected` 문장을 우선 사용한다.
3. `status: keep_distinct`면 각 표현을 별개 의미로 유지한다.
4. 항목이 없으면 새 conflict 블록을 추가한다.
5. `status: open`이어도 컴파일은 멈추지 않는다.
6. 대신 그 컴파일은 `warning 있는 컴파일`로 기록한다.
7. 가능하면 `targets`에 적힌 파일만 다시 쓴다.

### 질문을 남겨야 하는 경우

- 같은 역할을 `Product 기획자`, `PO`, `PM`처럼 다르게 쓸 때
- 같은 강점을 `로그 설계`, `데이터 분석`, `로그 기반 기획`처럼 섞어 쓸 때
- 같은 경력 범위를 `게임·플랫폼·부동산`, `게임과 앱`, `소셜 월드 플랫폼`처럼 달리 쓸 때

### 충돌 처리 원칙

- `approved`면 `selected`를 정규 표현으로 사용한다
- `keep_distinct`면 통합하지 않는다
- `open`이면 아직 선택이 없는 상태로 본다
- 사용자가 `wording-conflicts.md`를 고치면 그 항목의 `targets`만 다시 컴파일한다

## 진단과 후속 행동

컴파일은 요약만 만드는 작업이 아니라, 최근 기록을 해석해 현재 상태와 다음 행동까지 드러내는 작업으로 본다.

### 최근 진단

- 최근 3~7일 기록에서 문제 신호가 약한지 강한지 본다.
- 반복 패턴이 유지되는지, 초점이 바뀌었는지 본다.
- 단정 대신 `최근 기록 범위에서는`, `이 시점에는` 같은 표현을 쓴다.
- 결과는 `Wiki/Self/Current Diagnosis.md`에 모은다.

### 해야 할 일

- 원문에 TODO가 직접 있으면 `Wiki/Self/TODO.md`에 옮긴다.
- 진단에서 바로 행동으로 바뀔 수 있는 항목이 있으면 TODO로 등록한다.
- 할 일은 짧고 검증 가능한 문장으로 쓴다.

### 공부한 개념과 더 생각할 것

- 링크나 메모 안에 공부한 개념이 있으면 한 번 더 짧게 정리한다.
- 그 개념이 현재 일, 삶, 프로젝트와 어떻게 연결되는지 한 줄 이상 남긴다.
- 이어서 생각해볼 질문이 있으면 `Wiki/Questions/*.md`에 남긴다.

## 파일명 호환 규칙

- `Inbox`, `Daily`, `Wiki`, `site`, `Meta` 아래 새 파일을 만들 때는 Windows 금지 문자 `<>:"/\|?*`를 쓰지 않는다.
- 파일명 끝의 점과 공백도 금지한다.
- 새 파일을 커밋하기 전에는 `scripts/check-windows-paths.sh --staged`를 실행한다.
- 이 저장소에서는 `.githooks/pre-commit`이 같은 검사를 자동으로 수행한다.

## 컴파일 예시

### 1. 욕구와 방향

원본:
[나는 무엇일까.txt](/Users/user/Documents/loglife/loglife-vault/Inbox/Text/나는%20무엇일까.txt)

추출:

- 성공, 자유, 인정 욕구가 함께 등장한다.
- 창업 욕망이 자기 방향과 연결된다.
- 리더십은 성격 문제라고 스스로 해석한다.

출력:

- `Wiki/Self/Profile.md`
- `Wiki/Self/Tensions.md`
- `Wiki/Self/Open Questions.md`

### 2. 성격과 대인관계

원본:
[나의 성격.txt](/Users/user/Documents/loglife/loglife-vault/Inbox/Text/나의%20성격.txt)

추출:

- 무뚝뚝함, 감정 거리두기, 침묵 내성
- 멘탈 강함으로 이어지는 장점
- 공감 부족과 사람을 다루는 어려움

출력:

- `Wiki/Self/Profile.md`
- 필요 시 `Wiki/Themes/관계.md`

### 3. 건강과 행동 변화

원본:

추출:

- 체중 증가라는 부작용
- 건강 판단 기준에 대한 혼란

출력:

- `Wiki/Themes/건강.md`
- 필요 시 `Wiki/Self/Tensions.md`

### 4. 생각의 전환

원본:
[생각의 변화.txt](/Users/user/Documents/loglife/loglife-vault/Inbox/Text/생각의%20변화.txt)

- 제도 해석이 소비 태도를 바꿨다.
- 생각 변화가 편견을 만든다는 자각이 있다.
- 자기 사고가 어떻게 틀어질 수 있는지 메타 인식이 있다.

출력:

- `Wiki/Themes/생각의-변화.md`
- `Wiki/Self/Profile.md`

## 링크가 있는 텍스트 처리

텍스트에 링크가 있으면 아래 순서로 다룬다.

1. 원문 링크를 보존한다.
2. 링크 내용을 확인한다.
3. 무엇에 대한 링크인지 한 줄로 요약한다.
4. 핵심 포인트를 2~4줄로 적는다.
5. 왜 내 기록과 연결되는지 한 줄 남긴다.
6. 관련 `Daily` 또는 `Wiki` 문서에 연결한다.

## 좋은 컴파일의 기준

- 한 번의 기록이 아니라, 시간이 쌓일수록 더 선명해져야 한다.
- 새 문서를 계속 늘리기보다 기존 노드를 업데이트해야 한다.
- 원본을 읽지 않아도 맥락이 보이되, 원본으로 역추적 가능해야 한다.
