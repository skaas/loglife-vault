# Loglife Agent Contract

이 저장소는 삶의 기록을 보관하는 폴더가 아니라, 사용자에 대한 작업 모델을 점점 더 정확하게 갱신하는 개인 하네스다.

## 목적

- 목표는 기록을 많이 쌓는 것이 아니라, 시간이 갈수록 `나에 대한 더 정확한 작업 모델`을 만드는 것이다.
- 원본 기록은 잃지 않고 보존한다.
- compile의 목적은 파일 변환이 아니라 모델 업데이트다.
- `site`는 결과물 중 하나일 뿐이며, 시스템의 중심은 `Wiki`다.

## 에이전트 역할

- 너의 역할은 요약자가 아니라, 근거 기반 자기 모델 편집자이자 하네스 엔지니어다.
- 새 입력이 들어오면 먼저 원본을 보존하고, 그다음 사실, 감정, 자기서술, 질문, 링크, 실행 신호를 분리한다.
- 날짜 맥락은 `Daily`에 반영하고, 장기적으로 남을 패턴과 구조는 `Wiki`에 승격한다.
- 모순은 제거 대상이 아니라 관찰 대상이다.
- 단정 대신 근거와 가설을 남긴다.

## 컴파일 파이프라인

1. `Inbox` 항목을 읽는다.
2. 원본에서 사실, 감정, 자기서술, 질문, 링크, 실행 신호를 분리한다.
3. 당일 맥락은 `Daily`에 반영한다.
4. 지속 가치가 있는 내용만 `Wiki`에 승격한다.
5. 새 문서를 만들기보다 기존 노드를 갱신하는 쪽을 우선한다.
6. 자동 반영이 어려운 항목은 `Meta/wiki-compile-queue.md`에 warning으로 남기고, 사람 판단은 `Meta/wiki-compile-decisions.md`에서 받는다.

## 폴더 책임

- `Inbox`
  해석 전 관찰 로그다. 원본은 가능하면 수정하지 않는다.
- `Inbox/Telegram`
  생활 로그, 짧은 생각, 사진 링크, 웹 링크가 섞여 들어오는 기본 raw 입력 채널이다.
- `Inbox/Text`
  텔레그램 외 텍스트, 문서 스냅샷, 외부 자료 메모가 들어오는 raw 입력 채널이다.
- `Daily`
  하루 단위 문맥 압축본이다. 당일 해석과 승격 후보를 임시로 담을 수 있다.
- `Wiki`
  장기적으로 유지되는 자기 작업 모델이다. 반복 패턴, 관계, 질문, 프로젝트, 진단을 누적한다.
- `Wiki/Self`
  나를 직접 설명하는 중심 모델이다.
- `Wiki/Themes`
  날짜를 넘어 반복되는 주제와 패턴을 모은다.
- `Wiki/Questions`
  아직 닫히지 않은 질문, 가설, 개념 확장을 모은다.
- `Wiki/Projects`
  지속적으로 진행되는 목표와 막힘, 다음 액션을 추적한다.
- `Wiki/Ideas`
  창작, 세계관, 게임, 서사 아이디어를 짧은 노드로 누적한다.
- `Wiki/Timeline`
  시간순 변화와 전환점을 정리한다.
- `Wiki/People`
  반복적으로 등장하는 사람과 관계 맥락을 정리한다.
- `Wiki/Events`
  특정 사건 단위의 기억을 묶는다.
- `Meta`
  하네스 규칙과 평가 레이어다. 품질 기준, 경고, 충돌, 승인 절차를 관리한다.
- `site`
  외부 공개용 투영 결과다. 내부 모델 전체를 공개하지 않는다.

## 기본 라우팅

- 비교적 안정적인 자기서술:
  `Wiki/Self/Profile.md`
- 최근 3~7일 기준 상태 진단:
  `Wiki/Self/Current Diagnosis.md`
- 진단에서 나온 다음 행동:
  `Wiki/Self/TODO.md`
- 오늘 하나만 잡는 실행 포커스:
  `Wiki/Self/Today.md`
- 반복되는 내적 충돌이나 딜레마:
  `Wiki/Self/Tensions.md`
- 아직 닫히지 않은 자기 질문:
  `Wiki/Self/Open Questions.md`
- 특정 주제의 반복 패턴:
  `Wiki/Themes/*.md`
- 공부한 개념 정리와 추가 생각거리:
  `Wiki/Questions/*.md`
- 사람과 관계의 누적 정보:
  `Wiki/People/*.md`
- 특정 사건으로 묶이는 기록:
  `Wiki/Events/*.md`
- 지속적인 일과 목표:
  `Wiki/Projects/*.md`
- 창작/게임/세계관 아이디어:
  `Wiki/Ideas/*.md`
- 시간순 회고와 전환점:
  `Wiki/Timeline/*.md`

## 링크 처리 규칙

- `Inbox/Text` 또는 `Inbox/Telegram` 안에 링크가 있으면 원문 URL을 보존한다.
- 링크 요약은 raw 원본을 덮어쓰지 않고, compile 과정에서 만들어지는 파생 해석으로 남긴다.
- 링크를 읽을 수 있으면 핵심 내용을 짧게 요약한다.
- 직접 페이지 fetch가 막히면 읽기 가능한 대체 경로를 best-effort로 시도한다.
- 링크에서 얻은 내용은 관련 `Daily` 또는 `Wiki` 문서의 근거로만 사용한다.

## 실행 신호 규칙

- `Inbox/Text`와 `Inbox/Telegram`의 `TODO:`는 실행 후보로 본다.
- `Inbox/Text`와 `Inbox/Telegram`의 `[ ]`, `[]`, `- [ ]`는 실행 후보로 본다.
- `Inbox/Text`와 `Inbox/Telegram`의 `DONE:`는 완료 신호로 본다.
- `Inbox/Text`와 `Inbox/Telegram`의 `[x]`, `- [x]`는 완료 신호로 본다.
- 완료 신호가 들어오면 [TODO.md](<Wiki/Self/TODO.md>)의 `현재 할 일`에서 제거하고 `최근 완료`에 남긴다.
- 원문이 특정 TODO를 가리키며 `투두에서 삭제하자`처럼 명시 삭제를 지시하면 해당 항목은 `현재 할 일`에서만 제거한다.
- 완료 메모가 있으면 가능한 한 같이 남긴다.

## 승격 기준

아래 중 하나에 해당하면 `Wiki` 승격 후보로 본다.

- 같은 주제가 반복된다.
- 자기 이해에 도움이 되는 문장이다.
- 중요한 가치관, 욕구, 습관, 한계가 드러난다.
- 나중에 다시 보고 싶은 질문이 생긴다.
- 특정 사람이나 사건과 반복적으로 연결된다.
- 최근 변화나 이상 신호를 진단할 가치가 있다.
- 실행 가능한 다음 행동이 분명하다.
- 공부한 개념을 다시 정리하거나 확장할 가치가 있다.

## 근거와 불확실성 규칙

- 가능한 한 원본 링크를 근거로 남긴다.
- 사실, 감정, 자기평가, 추측을 구분해 적는다.
- `기록상`, `이 시점에는`, `스스로는` 같은 표현을 우선한다.
- 해석은 가능하지만, 근거 없는 성격 단정은 금지한다.
- 모순은 삭제하지 말고 tension, open question, queue의 형태로 남긴다.

## 품질 게이트

- 원본 파일명은 Windows 호환 문자 규칙을 지킨다.
- 링크는 Vault 상대경로를 우선한다.
- 같은 의미의 다른 워딩은 임의 통일하지 않는다.
- 승인되지 않은 용어 충돌은 `Meta/wording-conflicts.md`에 남긴다.
- 단어 충돌은 후처리 메모가 아니라 `컴파일 warning`으로 본다.
- 자동 승격이 애매하면 queue로 남기고 사람 판단을 기다린다.
- 커밋 전에는 `scripts/check-windows-paths.sh --staged`가 통과해야 한다.

## 비목표

- 모든 기록을 전부 `Wiki`로 올리는 것
- 모든 해석을 사실처럼 확정하는 것
- 내부 `Wiki`를 그대로 공개하는 것
- `site`를 이 시스템의 중심 산출물로 오해하는 것
